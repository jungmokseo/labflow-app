## 1. AI 이중 처리 안티패턴

### 문제란?

여러 AI 모델을 연결하는 파이프라인에서, **이미 완성된 결과를 다른 AI가 불필요하게 다시 처리**하는 패턴이다.

이번 세션의 구체적 사례:

```
[이메일 브리핑 흐름 — 수정 전]

Gmail API → Gemini Flash (분류) → Sonnet (서사형 마크다운 생성)
                                          ↓
                                   ✅ 완벽한 마크다운
                                          ↓
                            Gemini Flash (brain.ts "자연스럽게 재구성")
                                          ↓
                                   ❌ 줄글로 변환됨
```

Sonnet이 줄 단위로 깔끔하게 포맷팅한 브리핑을, Gemini의 시스템 프롬프트에 있는 "내용을 자연스럽게 재구성하여 전달하세요"라는 지시 때문에 줄글로 뭉개버린 것이다.

### 왜 발생하는가?

- 파이프라인이 **범용 구조**로 설계되어 있기 때문이다. brain.ts의 Gemini는 모든 intent의 최종 응답을 생성하도록 설계되었는데, 일부 intent는 이미 AI가 완성된 결과를 만든 상태였다.
- 기능이 하나씩 추가되면서 파이프라인의 전체 흐름을 재검토하지 않아 누적된 문제다.

### 해결 방법

**Passthrough 패턴**: 특정 intent는 최종 AI 단계를 건너뛰고 결과를 직접 반환한다.

```
[수정 후]

Sonnet → 완성된 마크다운 → 사용자에게 직접 전달 (Gemini 스킵)
```

```typescript
const directPassthroughIntents = ['email_briefing', 'daily_brief', 'emerge', 'weekly_review'];
if (directPassthroughIntents.includes(intent) && directResult) {
  // Gemini를 거치지 않고 직접 스트리밍
  return directResult;
}
```

### 교훈

- AI 파이프라인에서는 **각 단계의 입출력 품질을 추적**해야 한다. 중간 결과가 이미 최종 품질이면 추가 처리는 오히려 품질을 떨어뜨린다.
- "범용 파이프라인 + 예외 처리" vs "intent별 전용 파이프라인" 사이의 트레이드오프를 인식해야 한다.

## 2. SSE 스트리밍과 진행 상태 표시

### SSE란?

Server-Sent Events — 서버가 클라이언트에게 **한 방향으로** 실시간 데이터를 보내는 HTTP 기반 프로토콜이다. WebSocket보다 단순하고 HTTP 인프라와 호환성이 좋다.

LabFlow의 Brain 채팅은 SSE로 세 가지 유형의 이벤트를 전송한다:

| 이벤트 타입 | 용도 | 예시 |
|-----------|------|------|
| `progress` | 처리 단계 표시 | "Gmail에서 이메일을 가져오고 있습니다..." |
| `token` | AI 응답 토큰 스트리밍 | "이번 주 요약..." (한 글자씩) |
| `done` | 완료 + 최종 결과 | 전체 응답 + 메타데이터 |

### 프론트엔드에서의 상태 관리

```
[상태 전이]

초기 → loading=true, thinkingSteps=[]     → "생각 중..."
     → progress 이벤트 수신               → "이메일을 확인하고 있습니다..."
     → progress 이벤트 추가 수신           → 이전 단계 ✅, 현재 단계 ⋯
     → token 이벤트 시작                   → 스트리밍 텍스트 표시 + 진행 단계 계속 표시
     → done 이벤트                        → 최종 메시지로 교체
```

이번 세션에서 수정한 핵심: 토큰 스트리밍이 시작되면 진행 단계가 사라지는 문제. `loading && !streamingContent` 조건을 `loading && thinkingSteps.length > 0`으로 변경하여, 스트리밍 중에도 진행 단계를 계속 표시하도록 했다.

## 3. Cold Start와 Retry 전략

### Cold Start란?

Railway 같은 컨테이너 서비스에서 서버가 일정 시간 요청이 없으면 **컨테이너를 내려놓고**, 다음 요청이 오면 다시 시작하는 것이다. 시작에 수 초~수십 초가 걸리므로 첫 요청이 실패할 수 있다.

### 기존 문제

```
시도 1: fetch() → 실패 (서버 아직 시작 안 됨)
        ↓ 1500ms 대기
시도 2: fetch() → 실패 (아직 시작 중)
        ↓ 에러: "서버에 연결할 수 없습니다"
```

### 수정된 전략

```
시도 1: fetch() → 실패
        ↓ /health 핑 전송 (웨이크업)
        ↓ 2500ms 대기
        ↓ 사용자에게 "서버에 다시 연결하고 있습니다... (1/2)" 표시
시도 2: fetch() → 실패 (아직 시작 중일 수 있음)
        ↓ /health 핑 전송
        ↓ 5000ms 대기
        ↓ 사용자에게 "서버에 다시 연결하고 있습니다... (2/2)" 표시
시도 3: fetch() → 성공 (서버 준비 완료)
```

**점진적 백오프(Progressive Backoff)**: 재시도 간격을 점점 늘려서, 서버가 준비될 시간을 확보하면서도 불필요한 대기를 최소화한다.

## 핵심 용어 정리

| 용어 | 설명 |
|------|------|
| SSE (Server-Sent Events) | 서버→클라이언트 단방향 실시간 이벤트 스트리밍 프로토콜 |
| Cold Start | 유휴 상태의 서버 컨테이너가 요청 시 재시작되는 현상 |
| Progressive Backoff | 재시도 간격을 점진적으로 늘리는 전략 (0→2.5s→5s) |
| Passthrough 패턴 | 파이프라인의 특정 단계를 조건부로 건너뛰는 설계 패턴 |
| Double Processing | 이미 처리된 결과를 불필요하게 다시 처리하는 안티패턴 |
| Shadow Session | Brain 채팅에서 도구(이메일, 캘린더) 호출 결과를 별도 채널에 저장하는 구조 |

---

## 더 알아보기

- **SSE vs WebSocket**: SSE는 단방향, WebSocket은 양방향. 채팅 앱에서 서버→클라이언트 스트리밍만 필요하면 SSE가 더 적합하다.
- **LLM 체이닝 패턴**: LangChain, LlamaIndex 등의 프레임워크에서 사용하는 체인 설계 원칙. 각 단계의 역할을 명확히 하고, 불필요한 중간 처리를 줄이는 것이 핵심.
- **AI 모델 선택 전략**: Gemini Flash(빠르고 저렴) vs Sonnet(품질 우선). 용도에 따라 적합한 모델을 선택하는 것이 비용과 품질 모두에 영향을 준다.
