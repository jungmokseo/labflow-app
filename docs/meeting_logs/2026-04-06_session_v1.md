## 1. 논의 요약

교수님이 Brain 채팅에서 "3시간 이메일 브리핑 해줘" 메시지를 보낼 때 "서버에 연결할 수 없습니다" 에러가 반복 발생하는 문제를 스크린샷과 함께 보고했다. 동시에 이메일 브리핑이 줄 단위 마크다운이 아닌 줄글로 출력되는 문제, 그리고 처리 진행 상태가 채팅에 표시되지 않는 문제도 함께 제기했다.

원인 분석 결과, 세 가지 문제의 근본 원인을 발견했다:

1. **서버 연결 실패**: Railway cold start 시 retry 딜레이(1500ms)가 부족하여 2회 시도 모두 실패
2. **이메일 브리핑 줄글**: Sonnet이 완벽한 마크다운을 생성한 뒤, brain.ts의 Gemini가 "자연스럽게 재구성"하면서 줄글로 변환 (핵심 발견)
3. **진행 상태 미표시**: UI에서 `loading && !streamingContent` 조건으로 토큰 스트리밍 시작 시 thinking steps가 사라짐

특히 이메일 브리핑 문제를 계기로 **Gemini 이중 처리 패턴 전체 감사**를 실시했다. `daily_brief`, `emerge`, `weekly_review`도 동일한 패턴(AI가 포맷팅한 결과를 Gemini가 다시 씹음)이 발견되었고, 모두 수정했다.

교수님의 확인으로 `weeklyReview`를 Sonnet으로 업그레이드하기로 한 이전 결정이 반영 안 된 것도 발견하여 함께 수정했다.

## 2. 결정 사항

| # | 결정 사항 | 배경/이유 |
|---|----------|----------|
| 1 | email_briefing, daily_brief, emerge, weekly_review 4개 intent는 Gemini 재처리 스킵 | AI가 이미 완성한 마크다운을 Gemini가 줄글로 변환하는 문제 근본 해결 |
| 2 | weeklyReview를 Gemini → Sonnet으로 업그레이드 | 이전 세션에서 합의했으나 미반영. Gemini fallback 유지 |
| 3 | 서버 연결 retry를 2회→3회, 딜레이 점진적 백오프(0/2.5s/5s)로 강화 | Railway cold start 시간이 1.5초보다 긴 경우 대응 |
| 4 | retry 시 /health 엔드포인트에 wake-up ping 전송 | cold start 웨이크업 가속 |
| 5 | thinking steps를 토큰 스트리밍 중에도 계속 표시 | 사용자가 처리 진행 상황을 볼 수 있도록 |

## 3. 변경 사항

이전 계획/구현 대비 변경된 것들:

- **brain.ts 아키텍처 변경**: `directPassthroughIntents` 배열 도입 — 특정 intent는 Gemini 최종 응답 생성을 완전히 우회하고 도구 결과를 직접 스트리밍
- **weeklyReview 모델 변경**: Gemini Flash → Sonnet (claude-sonnet-4) 우선, Gemini fallback
- **retry 정책 변경**: 고정 1500ms 1회 → 점진적 백오프 [0, 2500, 5000ms] 3회
- **timeout 변경**: 90초 → 120초

## 6. 미결 사항 / 열린 질문

- `emerge` (숨겨진 연결 발견)도 Sonnet으로 업그레이드할지는 미결정. 현재 Gemini 1회 처리로 유지.
- `dailyBrief`도 마찬가지. 데이터가 쌓이면 품질 평가 후 결정.
- 이메일 브리핑에서 이모지 사용 여부 — email.ts의 Sonnet 프롬프트에 "이모지 절대 사용 금지" 지시가 있으나, brain.ts 시스템 프롬프트에는 "이모지 사용하라"는 지시도 혼재. 현재 Sonnet 직접 반환으로 일관성 확보.
